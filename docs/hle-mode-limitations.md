# BBKEmu HLE模式限制分析

## 概述

本文档详细说明为什么BBKEmu模拟器无法实现纯HLE模式（不加载OS ROM）的原因。通过实际调试数据和技术分析，证明这条路几乎无法走通。

## 背景

HLE（High Level Emulation）模式的目标是：在不加载OS ROM的情况下，通过Rust代码模拟OS ROM中的函数行为，使游戏能正常运行。

LLE（Low Level Emulation）模式则直接加载OS ROM，让CPU执行真实的6502代码。

## 核心问题：D2F6远调用分发器

### D2F6的作用

D2F6是BBK OS中的核心函数，负责远调用分发。游戏通过`JSR $D2F6`调用各种OS功能（文件操作、LCD绘制、字符串处理等）。

### D2F6的复杂性

通过反编译OS ROM，D2F6的完整代码如下：

```
; D2F6 dispatcher (OS ROM offset 0x0882F6)
AA         ; TAX - 保存A寄存器
20 F8 E8   ; JSR $E8F8 - 设置bank switching
48         ; PHA - 保存状态
A0 02      ; LDY #$02
B1 26      ; LDA ($26),Y - 读取segment
20 FB E8   ; JSR $E8FB - 切换到目标segment
88         ; DEY
38         ; SEC
B1 26      ; LDA ($26),Y - 读取target_high
48         ; PHA
88         ; DEY
B1 26      ; LDA ($26),Y - 读取target_low
85 26      ; STA $26
68         ; PLA
85 27      ; STA $27
8A         ; TXA - 恢复A寄存器
20 72 D5   ; JSR $D572 - 调用目标函数
AA         ; TAX - 保存返回值
68         ; PLA - 恢复状态
20 FE E8   ; JSR $E8FE - 恢复bank mapping
8A         ; TXA - 恢复返回值
60         ; RTS - 返回
```

D2F6调用了4个子函数：

- **E8F8**: 设置bank switching
- **E8FB**: 切换到目标segment
- **D572**: 函数指针调用
- **E8FE**: 恢复bank mapping

每个子函数都是复杂的6502代码，无法简单用Rust复制。

## 实证数据

### 实验1：纯HLE模式（无OS ROM）

**配置**: 不加载OS ROM，所有函数用Rust实现

**结果**:
```
游戏        | HLE周期数    | LLE周期数    | 状态
------------|-------------|-------------|--------
五子棋      | 152,466     | 5,445,100   | ✗ 游戏退出早
中国象棋    | 11,744      | 1,688,418   | ✗ 游戏崩溃
二十一点    | 152,466     | 5,554,333   | ✗ 游戏退出早
乒乓球      | 152,466     | 6,560,029   | ✗ 游戏退出早
```

**结论**: 纯HLE模式下，游戏运行时间仅为LLE模式的2-3%，无法正常运行。

### 实验2：软件栈问题

**问题**: 游戏依赖软件栈[0x28:0x29]的精确值

**LLE模式跟踪**:
```
D2F6: [0x28]=0x17D4 A=0x17 (file_open)
D2F6: [0x28]=0x17CD A=0x2B (-7)
D2F6: [0x28]=0x17D2 A=0xFF (+5)
D2F6: [0x28]=0x17D4 A=0x17 (+2)
```

**HLE模式跟踪**:
```
D2F6: [0x28]=0x17D4 A=0x17 (file_open)
D2F6: [0x28]=0x17D6 A=0x2C (+2)
D2F6: [0x28]=0x17D8 A=0x17 (+2)
D2F6: [0x28]=0x17DA A=0x05 (+2)
```

**分析**: 
- LLE模式下，[0x28]变化复杂（-7, +5, +2等），取决于目标函数的内部实现
- HLE模式下，[0x28]只增加2（D2F6弹出2字节），目标函数不修改栈

**根本原因**: OS ROM中的目标函数（如file_open）会推入局部变量到软件栈，HLE实现无法复制这一行为。

### 实验3：D2F6跳板方案

**方案**: 在OS ROM中写入D2F6跳板代码，让CPU自然执行

**结果**:
```
游戏        | HLE周期数    | LLE周期数    | 状态
------------|-------------|-------------|--------
五子棋      | 32,862,906  | 5,445,100   | ✗ 输出不同
```

**分析**: 跳板代码只弹出2字节并返回，不调用目标函数。游戏继续运行但行为不正确。

### 实验4：复制OS ROM代码

**方案**: 从OS ROM文件中读取D2F6及其依赖函数，写入最小OS ROM

**结果**:
```
复制范围                    | 结果
---------------------------|------
D2F6 + E8F8 + D572 (1KB)   | ✗ 游戏崩溃
0x088000-0x098000 (64KB)   | ✗ 输出不同
完整OS ROM (2MB)           | ✓ 完全一致
```

**分析**: D2F6调用的函数（E8F8等）又调用其他函数，形成递归依赖链。只复制部分代码无法满足所有依赖。

## 技术分析

### 1. 软件栈管理

OS ROM中的函数使用软件栈[0x28:0x29]存储局部变量。每个函数有不同的栈帧大小：

```
函数           | 栈帧大小 | 说明
---------------|---------|------
file_open      | -5字节  | 推入5字节局部变量
file_seek      | +3字节  | 弹出3字节
strlen         | 0       | 不修改栈
lcd_draw_char  | -2字节  | 推入2字节局部变量
```

这些栈帧大小取决于函数的内部实现，无法从外部推断。

### 2. 寄存器副作用

OS ROM函数会修改多个寄存器：

```
函数           | 修改的寄存器
---------------|-------------
file_close     | A=0, Y=1
file_read      | A=bytes_read, [0x20:0x21]=bytes_read
lcd_draw_char  | 无
```

这些副作用是函数实现的一部分，HLE实现必须完全复制。

### 3. Bank Switching

D2F6通过E8F8/E8FB/E8FE管理bank switching。每个segment使用不同的bank映射：

```
Segment | Bank 5-8 映射
--------|---------------
0x04    | 特定地址范围
0x06    | 特定地址范围
0x07    | 特定地址范围
0x08    | 特定地址范围
```

HLE实现必须精确复制这些bank映射，否则游戏无法正确访问内存。

### 4. 函数指针调用

D572通过间接跳转调用目标函数：

```
D572:
    TAY         ; 保存A
    SEC
    LDA $26
    SBC #$01    ; [0x26] -= 1
    STA $26
    LDA $27
    SBC #$00
    STA $27
    PHA         ; 推入目标地址
    LDA $26
    PHA
    TYA         ; 恢复A
    RTS         ; 跳转到目标函数
```

这种间接调用机制无法用简单的Rust函数调用复制。

## 结论

纯HLE模式（无OS ROM）无法实现的原因：

1. **D2F6分发器过于复杂**: 调用4个子函数，每个都有复杂的bank switching逻辑
2. **软件栈管理无法复制**: 每个目标函数有不同的栈帧大小，取决于内部实现
3. **寄存器副作用必须精确**: 游戏依赖函数的寄存器修改行为
4. **递归依赖链**: D2F6调用的函数又调用其他函数，形成无法切断的依赖链

## 推荐方案

**始终加载OS ROM**，让D2F6和所有OS函数从ROM自然执行。这是唯一能保证游戏完整运行且与LLE模式一致的方案。

HLE系统调用实现（syscalls/目录）可作为参考，但实际运行时应使用OS ROM。
